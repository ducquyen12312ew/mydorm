## How to Get Each Key

### MongoDB URI

**Local MongoDB (development)**
```
MONGO_URI=mongodb://127.0.0.1:27017/Dormitory
MONGODB_URI=mongodb://127.0.0.1:27017/Dormitory
```

**MongoDB Atlas (production)**
1. Go to https://cloud.mongodb.com → sign in or create account
2. Create a free M0 cluster (or paid tier for production)
3. Click "Connect" → "Drivers" → select Node.js
4. Copy the connection string:
   ```
   mongodb+srv://<username>:<password>@<cluster>.mongodb.net/Dormitory?retryWrites=true&w=majority
   ```
5. Replace `<username>` and `<password>` with your database user credentials
6. **Important:** Set both `MONGO_URI` and `MONGODB_URI` to the same value

**Common Atlas setup mistakes:**
- Forgot to whitelist your IP: Cluster → Network Access → Add IP Address
- Password contains special characters — URL-encode them (e.g. `@` → `%40`)
- Wrong `authSource` — Atlas uses `authSource=admin` by default (already in SRV URI)

### Secure Random Secrets

For `SESSION_SECRET`, `MOBILE_JWT_ACCESS_SECRET`, `MOBILE_JWT_REFRESH_SECRET`, `QR_SECRET`:

```bash
# Generate a 64-byte hex secret (128 characters)
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"

# Or using openssl:
openssl rand -hex 64
```

**Each secret must be unique.** Never reuse secrets across different variables.

### Gmail SMTP (App Password)

1. Your Google account must have 2-Step Verification enabled
2. Go to: https://myaccount.google.com/apppasswords
3. Select "Mail" and your device type
4. Click "Generate" — you get a 16-character password
5. Use that as `SMTP_PASSWORD` — it is different from your Gmail login password

```
SMTP_USER=your-email@gmail.com
SMTP_PASSWORD=abcd efgh ijkl mnop   # (spaces are ignored by Gmail)
SMTP_FROM_EMAIL=your-email@gmail.com
```

### Twilio (SMS 2FA — optional)

1. Sign up at https://www.twilio.com
2. Go to Console → Account Info
3. Copy `Account SID` → `TWILIO_ACCOUNT_SID`
4. Copy `Auth Token` → `TWILIO_AUTH_TOKEN`
5. Get a phone number → `TWILIO_PHONE_NUMBER` (format: `+1234567890`)

If you don't configure Twilio, SMS 2FA is disabled. TOTP (Google Authenticator) still works.

## MongoDB Connection Audit

### Observed Behavior (tested 2026-05-28)

| Scenario | Behavior | Exit Code |
|----------|---------|-----------|
| Valid URI, correct credentials, DB exists | `Database Connected Successfully` | Continues normally |
| Valid URI, DB does not exist | `Database Connected Successfully` (MongoDB creates DB lazily) | Continues normally |
| Valid URI, wrong credentials | `[FATAL] MongoDB initial connection failed: Authentication failed.` | `process.exit(1)` |
| Invalid URI format (not mongodb:// or mongodb+srv://) | `[FATAL] MongoDB initial connection failed: Invalid scheme...` | `process.exit(1)` |
| Valid URI format, host unreachable | Silent — waits up to 30 s (serverSelectionTimeoutMS), then fires 'error' event | App keeps running without DB |

**Note:** The "unreachable host" scenario does NOT crash the process — the app starts but all database operations fail. In production, monitor `mongoose.connection.readyState` or use a health check endpoint.

### Connection Parameters

```javascript
// src/config/config.js
mongoose.connect(mongoUri, {
    serverSelectionTimeoutMS: 30000,  // how long to wait for initial server selection
    socketTimeoutMS: 45000,           // how long before a socket times out
});
```

### Test Commands

```bash

# Expected: { ok: 1 }

# 3. Test Atlas connection (replace URI)

```

### Atlas-Specific Issues

**IP Whitelist:** MongoDB Atlas rejects connections from non-whitelisted IPs.
- Development: add your IP at Cluster → Network Access → Add IP Address
- Production (dynamic IP / hosting): add `0.0.0.0/0` (allow all) or your server's static IP
- Warning: `0.0.0.0/0` means the DB is accessible from any IP — rely on username/password security

**SRV vs Standard URI:**
- SRV (`mongodb+srv://`): DNS-based, handles failover automatically. Use this for Atlas.
- Standard (`mongodb://host:27017`): explicit host/port. Use this for local.

**authSource:** Atlas uses `authSource=admin` by default (already in SRV URI). If you get "Authentication failed" with credentials you know are correct, verify the authSource matches where the user was created.

**Password special characters:** If your password contains `@`, `:`, `/`, `?`, `#`, `[`, `]`, `!`, or `%`, URL-encode them:
```
@ → %40    : → %3A    / → %2F    ? → %3F
```

## Security Recommendations

### Secret Generation

Every production deployment must generate fresh, unique secrets:

```bash
# Run once — copy output to .env
node -e "
const c = require('crypto');
console.log('SESSION_SECRET=' + c.randomBytes(64).toString('hex'));
console.log('MOBILE_JWT_ACCESS_SECRET=' + c.randomBytes(64).toString('hex'));
console.log('MOBILE_JWT_REFRESH_SECRET=' + c.randomBytes(64).toString('hex'));
console.log('QR_SECRET=' + c.randomBytes(64).toString('hex'));
"
```

### Generated Secret Policy

All secrets were generated with `crypto.randomBytes(64).toString('hex')` — 512 bits of entropy, 128 hex characters each. Verified unique (no two secrets share a value).

| Secret | Length | Entropy | Uniqueness |
|--------|--------|---------|-----------|
| `SESSION_SECRET` | 128 chars | 512 bit | ✅ unique |
| `MOBILE_JWT_ACCESS_SECRET` | 128 chars | 512 bit | ✅ unique |
| `MOBILE_JWT_REFRESH_SECRET` | 128 chars | 512 bit | ✅ unique |
| `QR_SECRET` | 128 chars | 512 bit | ✅ unique |

### Minimum Secret Length (enforced by validateEnv.js)

| Secret | Minimum | Recommended |
|--------|---------|-------------|
| `SESSION_SECRET` | 32 chars | 128 chars (hex) |
| `MOBILE_JWT_ACCESS_SECRET` | 32 chars | 128 chars (hex) |
| `MOBILE_JWT_REFRESH_SECRET` | 32 chars | 128 chars (hex) |
| `QR_SECRET` | 32 chars | 128 chars (hex) |

### Preventing .env Leaks

1. **Verify .gitignore:** `git check-ignore -v .env` should output `.gitignore:.env`
2. **Check git history:** `git log --all -- .env` — should return nothing
3. **Audit staged files:** `git status` before every commit — never stage `.env`
4. **Secret scanning:** GitHub's built-in push protection will block known credential patterns

### Production Secret Rotation

When rotating secrets:
1. Update the value in `.env` (or your secrets manager)
2. Restart the server
3. **Session:** All existing web sessions will become invalid (users logged out) — expected
4. **JWT refresh tokens:** All mobile refresh tokens will fail to verify — mobile users will be logged out
5. **QR:** All existing QR tokens will fail signature check — users regenerate on next card view
6. This is the correct behavior — secret rotation intentionally invalidates old sessions

### Production Checklist

Before deploying to production:

- [ ] `NODE_ENV=production`
- [ ] `SESSION_SECRET` set to a fresh 64-byte hex secret
- [ ] `MOBILE_JWT_ACCESS_SECRET` set to a fresh secret (≠ SESSION_SECRET)
- [ ] `MOBILE_JWT_REFRESH_SECRET` set to a fresh secret (≠ ACCESS_SECRET)
- [ ] `QR_SECRET` set to a fresh secret (≠ other secrets)
- [ ] `MONGO_URI` points to production database (Atlas or self-hosted)
- [ ] MongoDB IP whitelist includes your server's IP
- [ ] `SMTP_USER` + `SMTP_PASSWORD` configured (use Gmail app password, not login password)
- [ ] `SKIP_EMAIL=false` (emails actually sent)
- [ ] TLS termination at reverse proxy (nginx/Caddy) — app does not need `USE_HTTPS`
- [ ] `.env` not committed to git: `git check-ignore .env` should output the filename
- [ ] Default admin password changed from `Admin@123456` / `admin123`
- [ ] `REDIS_URL` configured if running multiple backend instances

## Runtime Startup Validation

`src/config/validateEnv.js` runs as the **very first module** on startup. It checks for missing, too-short, and insecure-default secrets before any database connection is attempted.

### Rules enforced

| Check | Dev behavior | Prod behavior |
|-------|-------------|--------------|
| `MONGO_URI` or `MONGODB_URI` missing | `process.exit(1)` | `process.exit(1)` |
| `SESSION_SECRET` missing or < 32 chars | warning | `process.exit(1)` |
| `MOBILE_JWT_ACCESS_SECRET` missing or < 32 chars | warning | `process.exit(1)` |
| `MOBILE_JWT_REFRESH_SECRET` missing or < 32 chars | warning | `process.exit(1)` |
| `QR_SECRET` missing or < 32 chars | warning | `process.exit(1)` |
| Any secret matches known insecure default | warning | `process.exit(1)` |
| ACCESS_SECRET === REFRESH_SECRET | warning | `process.exit(1)` |
| ACCESS_SECRET === QR_SECRET | warning | `process.exit(1)` |
| MONGO_URI invalid format | `[FATAL] process.exit(1)` | `[FATAL] process.exit(1)` |
| MONGO_URI auth failure | `[FATAL] process.exit(1)` | `[FATAL] process.exit(1)` |
| MONGO_URI unreachable | Hangs 30s, then error event | Hangs 30s, then error event |

## Known Deployment Risks

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Default JWT/QR secrets in production | 🔴 Critical | Generate and set all secrets before go-live |
| MongoDB IP whitelist not configured | 🔴 Critical | Add server IP in Atlas Network Access |
| `.env` accidentally committed | 🔴 Critical | Always check `git status` before committing |
| Social login (Google/Facebook) buttons shown but not functional | Medium | Remove buttons or show "coming soon" label |
| `forgot-password` renders success without sending email | Medium | Requires SMTP configuration to work |
| No TLS at app layer | Low | Terminate TLS at nginx/load balancer — do not disable HSTS |
| Unreachable MongoDB causes silent hang (no immediate crash) | Low | Add `/health` endpoint that checks `mongoose.connection.readyState` |
